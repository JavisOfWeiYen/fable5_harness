Repository Review

fable5_harness
Repo 評估報告

一套針對 Claude Code 設計的 Harness Engineering 工作制度， 透過角色分工、執行證據與獨立驗收，降低複雜開發任務中的誤判與「假完成」。

Repository：JavisOfWeiYen/fable5_harness 適用環境：Claude Code、WSL2、Linux、macOS 評估情境：獨立 WSL2 測試環境

一、這個 Repo 是什麼

fable5_harness 不是一般程式套件，也不是模型或 Agent Framework。 它比較像是一套安裝到 Claude Code 使用者環境中的「工程工作制度包」。

實際安裝的東西其實很少：一份精簡的 router、六份制度文件、三個自訂 agent，以及三個選配 hooks。 它不新增任何產品介面；真正改變的，是 Claude Code 在每次派工、驗收、卡關與收尾時的工作姿勢。

安裝後，主要內容會放入：

~/.claude/
主 Session 指揮化 主 Claude session 以拆解、派工與決策為主，避免自己承擔所有探索與實作。
完成必須有證據 不能只看程式碼或修改紀錄，就宣稱任務已經完成。
實作與驗收分離 由 fresh-context verifier 重新執行驗證，降低實作者自我確認偏誤。
核心價值 它將 Claude Code 從「會寫程式的助手」，轉變為更接近有明確分工、驗收與治理流程的 AI 工程團隊。

全域 CLAUDE.md

全域 CLAUDE.md 的定位不是塞入所有規則，而是作為一個精簡的 制度路由器：告訴 Claude 在不同情境下，應該讀取哪一份規範文件， 並維持少數不可被忽略的核心原則。

六份制度文件的分工

  • docs/10-model-dispatch.md：模型選擇、派工、升降級與驗證契約。
  • docs/20-judgment-rubrics.md：完成、升級、停下問人與方向判斷的判準。
  • docs/30-delegation-templates.md：五種常見任務的 prompt 骨架。
  • docs/40-maintenance-protocol.md：制度本身如何被安全更新。
  • docs/00-harness-diagnosis.md 與 docs/50-letter-to-future-sessions.md：退化模式診斷與跨 session 交接。

三個自訂 Agent

  • implementer：負責實作與自我測試，但不負責最終驗收。
  • verifier：只接收驗收條件與成果位置，以 fresh context 再次執行測試。
  • hard-solver：只有在一般策略已失敗、問題確實困難時才啟用。
真正生效的是 ~/.claude/ 內的 payload Repo 根目錄本身只是 installer package；安裝完成後可以刪除，實際發揮作用的是被複製進使用者環境的這些檔案。

三、它如何改變 Claude Code 的工作方式

安裝後，Claude Code 面對較複雜的任務，會傾向遵循以下流程：

使用者提出需求
↓
主 Session 拆解需求與定義驗收標準
↓
Implementer 或其他 Subagent 執行
↓
回報修改檔案、測試結果與執行證據
↓
Verifier 以 Fresh Context 再驗證
↓
通過後才能向使用者宣稱完成
「完成」的定義更嚴格 每一項驗收條件都需要對應的執行證據；重要工作還應接受獨立驗證，並檢查至少一項最可能發生的回歸問題。

幾條值得知道的具體規則

  • 長工作會出聲：預期超過幾分鐘的步驟，動手前先一行說明在做什麼、大約多久、Esc 可中斷；階段切換回報一行，等待 API 重試也會明說，不會沉默等待。
  • 互動風格可選：安裝時選「全自主」（預設，可逆步驟直接做）或「節點回報」（多分鐘工作先給 ≤5 行計畫等 OK、階段間一行回報、回覆結論先行）；兩種風格下失敗與風險都完整報告。
  • 批量編輯有門檻：主對話 inline 編輯超過 2 個檔案（或約 40 行 diff）就必須把逐字 hunks 交給 implementer 執行——「已經知道確切內容」正是該委派的訊號，不是豁免理由。
  • 多項查找要逐項證據：查 N 個變數／事實時，每一項都要附 file:line 與該行原文引用、沒查到的分開列——猜的引不出原文，當場露餡。

四、安裝之前的三個根因

要理解上面的流程解決了什麼，得先看沒有制度時會出什麼問題。 問題不在於模型完全不能做事，而在於它會在幾個固定位置用錯工作方式。

01

主對話自己下場做粗活

整檔讀大檔、反覆看截圖、批次改檔都塞進主對話。context 撐爆後觸發壓縮，模型開始忘記前面講好的限制。

02

沒有常駐制度

每個新 session 都重新摸索怎麼跑、怎麼驗、你偏好什麼；學到的教訓沒有固定去處。

03

完成由模型自己說了算

「程式看起來沒問題」不是證據。沒跑測試、沒打開頁面、沒讀回檔案，就只是宣稱。

五、六個關鍵時刻被改寫

Harness 的價值不在新功能，而在把上述容易失手的時刻，變成有規則、有證據、有交接的流程。 以下六個時刻，各以「無 harness」與「有 harness」對照，並附上實際生效的制度片段。

01 · 開場

無 harness

session 是白紙。模型靠直覺猜這台機器、這個使用者、這個 repo 的工作方式。

有 harness

根 router 自動載入：觸發派工、驗收、維護、複雜任務時，先讀對應制度文件。

02 · 主對話

無 harness

主對話是工人：親自掃 repo、親自讀長檔、親自跑迭代。context 被拿去當工作台。

有 harness

主對話是指揮官：粗活派給 subagent，回來只收結論、證據與 file:line。

03 · 完成

無 harness

「我實作了，所以應該會動。」沒有瀏覽器、沒有測試、沒有 read-back。

有 harness

完成必須有執行證據；實質工作由 fresh-context verifier 驗收，做的人不自驗。

04 · 卡關

無 harness

同一招重試第五次，越修越大，卻說不清楚為什麼這一步還服務目標。

有 harness

升級與換路都有條件：同一子任務錯兩次升級；第三輪必須換模型、換拆法或問使用者。

  • sonnet日常明確任務。
  • opus同一子任務失敗兩次，附完整失敗軌跡。
  • hard-solver常規嘗試卡死後的 max effort 最後手段。
  • user品味題、破壞性操作、真正有歧義的選擇。

05 · 紀律

無 harness

規則寫在文件裡，但模型略讀後照舊。制度靠自律，最容易退化。

有 harness

once-per-session hooks 在第一次派工前注入提醒；做過事（派過工或改過檔）的 session 第一次收工前注入 done-check——把最高價值觸發變成機制，純問答 session 零成本。

06 · 記憶

無 harness

你糾正過的事，只活在這個 session。下個 session 又踩一次。

有 harness

教訓有去處：專案事實進 memory，跨專案 doctrine 進 docs，handoff 進 logbook；改規則先問人。

六、一次實測對照

同一份 485 行的完整需求（Apex Sprint Challenge，F1 風格賽車遊戲，內含 14 條指定測試）， 沒裝與裝了 harness 的環境各從零做一次。兩個成品連同原始 prompt 收錄在本 repo 的 benchmark/，並由兩個互不知情的 agent 用逐字相同的協議實測（載入、實駕、 完整 demo 賽跑完、tests.html、逐項需求對照）。單一任務、單次執行——這是可重驗的切片， 不是統計結論。

關於上一次的一次性實驗 本節第一版引用的是更早的一次性實測（彈珠台）。那批產物後來在深度 review 中被指出 未收錄進 repo、結論無法重驗——恰好踩中本制度自己的紅線「有結論、沒證據」，於是整包 刪除、在乾淨環境重測。現在的數據全部來自可重驗的 benchmark/。這個教訓 本身比舊數據更有價值：連自己的行銷報告，也要過得了自己的驗收規則。

這個結果怎麼讀

成品分數幾乎打平——這不是意外，而是校準。這份 prompt 自帶 485 行規格與 14 條驗收測試， 等於把 harness 的核心（明確的驗收條件）預先搬進了任務本身；在這種任務上，沒制度的模型 本來就該做得好。差異出現在制度性的痕跡：獨立重驗腳本、自我申報的未驗證項、集中設定—— 這些不是「這次跑分」的加分，而是「下次要改」時的保險。

所以這個切片誠實支持的是本報告一貫的定位：harness 墊高的是下限與可稽核性， 不是能力天花板。它的價值要在規格不完整、驗收沒人幫你寫、一做就是幾天的日常任務上才拉開 ——也就是第五節那六個時刻描述的情境。

最鋒利的差異只有一句話 當 prompt 自帶制度時，兩邊都好——但日常任務的 prompt，沒有 485 行。

七、安裝機制的優點

這個 Repo 並不是單純提供一組檔案讓使用者直接複製。它刻意把安裝本身也設計成一個受控流程。

  • 先檢查既有的 ~/.claude/CLAUDE.md、docs 與 agents。
  • 修改前先備份現有檔案。
  • 發現同名檔案衝突時停止，不直接覆蓋。
  • 既有 CLAUDE.md 採合併策略，而不是整份取代。
  • 安裝時詢問成本偏好、主要用途、回覆語言、互動風格（全自主／節點回報）、是否有 Fable 5 權限，以及 hooks 設定。
  • 已安裝的機器由 upgrade_harness.md 升級：逐 hunk 移植、不覆蓋個人化；安裝時記錄 package commit，升級可精準 diff。
  • 偵測 WSL、Linux 工具與測試環境，避免殘留 placeholder。
不要使用整批 cp 直接覆蓋 正確方式應遵循 Repo 提供的安裝流程，尤其 hooks.json 應合併進既有設定， 不應直接覆蓋 ~/.claude/settings.json。

安裝範圍：同一個使用者環境只需一次

Harness 安裝在 Linux 使用者層級的 ~/.claude/，因此同一個 WSL2 instance、 同一個 Linux 使用者只需要安裝一次。之後從該環境進入不同專案並啟動 Claude Code， 都會共用這套全域制度；各專案只需另外提供自己的專案層級 CLAUDE.md。

何時才需要重新安裝 換成另一個 WSL2 instance、另一個 Linux 使用者，或刪除並重建原本環境時，才需要再次安裝。

八、選配 Hooks 的作用

Hooks 需要 jq，主要支援 WSL、Linux 與 macOS。它們的目的不是增加功能， 而是強制工作流程在重要節點進行自我檢查。

  • 第一次派出 subagent 前，提醒 Claude 讀取派工制度。
  • 進行過實質工作後，在 session 首次準備結束時，要求重新檢查完成標準。

使用者可見的行為是以上兩種；第三個 hook 沒有輸出，只在 Claude 改動檔案時默默標記 「這個 session 做過實質工作」，供第二項判斷是否需要收尾檢查——純問答 session 因此零成本。

Session marker 會放在：

~/.cache/claude-doctrine/

從設計目的來看，hooks 不只是「提醒」，而是把流程治理嵌入 Claude Code 的執行生命週期。

九、需要注意的成本與風險

1. Token 成本與執行時間會提高

一項工作可能經歷「實作」與「獨立驗證」兩次模型執行；若使用高階模型與高 effort， 成本與等待時間都會明顯增加。

2. 小任務可能變得過重

強制派工與驗收很適合大型、風險高或容易誤判的任務，但對改一行文字、 查一個設定等簡單工作，流程成本可能大於實際收益。

3. 模型與 Agent 格式可能隨版本變化

Claude Code 支援的模型 alias、effort 值與 agent frontmatter 可能改變， 安裝時應重新驗證，不能長期假設既有設定永遠有效。

4. 目前不是一鍵式 Plugin

這套制度依賴全域 ~/.claude/CLAUDE.md、環境偵測與設定合併， 因此本質上比一般 plugin 更接近一套「使用者層級的工程治理配置」。

5. 誠實的邊界：制度做不到的事

  • 不提高能力天花板。遇到超出模型推理上限的題，再多流程也不會把它變成更大的模型。
  • 不補品味。視覺方向、文風、兩個都不錯選哪個，制度要求模型提出 2–3 個候選並交給你決定。

換句話說，制度的價值有一半在於承認邊界：它提升的是工作的下限與可驗證性，而不是能力的上限。

實務風險 若規則過度僵化，Claude 可能為了符合流程而製造不必要的 agent、驗證與報告， 因此需要依任務風險調整制度強度，而不是所有任務一體適用。

十、簡單任務如何避免「殺雞用牛刀」

對簡單任務，最好的做法通常不是完全停用 Harness，而是改用 lightweight inline mode：由主 session 直接處理，不派出 implementer， 也不啟動 fresh-context verifier，但仍保留最小必要驗證。

適合輕量模式的任務

  • 讀取已知檔案的一小段內容。
  • 執行一次明確且範圍有限的搜尋或命令。
  • 修改一個已經理解的檔案，例如錯字、CSS 間距或單一設定值。
  • 純問答、程式碼解釋或錯誤訊息分析，且不涉及實際修改。

Small inline task 的判斷原則

依此 Repo 的工作判準，通常需同時符合以下條件：

  1. 只修改一個檔案。
  2. 修改規模不超過約 50 行。
  3. 不是將被正式執行、發布或高度依賴的重要成果。

即使修改行數很少，只要涉及部署、正式 API、權限、安全性、資料遷移或高風險腳本，仍應採完整驗證流程。

建議直接使用的指令

這是小型任務，使用 lightweight inline mode：
- 由主 session 直接完成
- 不派 subagent
- 不啟用 implementer
- 不啟用 fresh-context verifier
- 只做最小必要修改
- 完成後執行最小相關檢查並回報結果
任務類型 建議模式 最低驗證方式
修正文案錯字、格式或單一 CSS 間距 Inline Read-back 或重新載入頁面
修改一個變數名稱或單一設定值 Inline 執行最相關的單元測試或設定檢查
純問答、程式碼說明、錯誤訊息分析 Direct Q&A 通常不需要實作型驗證
修改部署腳本、正式 API 或安全性設定 Full Harness 實際執行測試加獨立 verifier
跨模組除錯、新功能或多檔案修改 Full Harness 驗收條件、執行證據與回歸測試
關鍵原則 輕量模式不是完全跳過品質檢查，而是把「多 Agent 獨立驗證」縮減為「主 session 的最小必要驗證」。

什麼時候才需要完全停用

只有在要做安裝前後的基準比較、排查 Harness 本身是否干擾行為，或進行完全無制度的對照實驗時， 才建議暫時停用全域規則： 

mv ~/.claude/CLAUDE.md ~/.claude/CLAUDE.md.disabled
# 重新啟動 Claude Code 後進行測試

mv ~/.claude/CLAUDE.md.disabled ~/.claude/CLAUDE.md
Hooks 需另外處理 若安裝時已啟用 hooks，單純改名 CLAUDE.md 不會停用 hooks； 還需暫時移除或停用 ~/.claude/settings.json 中對應設定。日常小任務不建議採用這種方式。

十一、在獨立 WSL2 環境中測試的意義

在任一 WSL2 instance 的 Linux 使用者環境中安裝後，檔案會位於： 

$HOME/.claude/

這些設定只會影響該 Linux 使用者環境內執行的 Claude Code，不會自動套用到其他 WSL2 instance。 因此，乾淨且獨立的 WSL2 instance 很適合作為可刪除、可重建的隔離測試環境。

建議安裝流程

結論

概念完整，對既有設定的保護也相當仔細，適合先在獨立的 WSL2 測試環境中驗證。

它最有價值的地方，不是增加更多 prompt，而是建立一套可被遵循、檢查與驗證的 AI 工程工作制度。對長時間、跨檔案、多步驟、容易誤判的開發任務而言， 這類 Harness 能顯著提升可信度。

真正的代價是 token 成本、執行時間，以及部分簡單任務可能因強制派工與驗證而顯得過重。 最理想的使用方式，是依任務風險分級：高風險任務完整套用，低風險任務採精簡流程。

歸根究柢，裝與不裝之間，模型本身沒有變聰明；改變的是：context 不再被粗活耗盡、 「完成」重新具備份量、判斷題有了成文的答案，而你糾正過的教訓會被保留下來。

十三、建議的導入策略

  1. 先在獨立 WSL2 環境測試：避免直接改動主要工作環境。
  2. 成本偏好先選中間檔：先觀察成本與驗證品質，不要一開始就把所有工作推到最高 effort。
  3. 挑三種任務驗證：小修改、中型功能、跨模組除錯各測一次。
  4. 記錄額外成本：比較安裝前後的 token、時間、返工次數與錯誤率。
  5. 建立風險分級：決定哪些任務需要 verifier，哪些可由主 session 直接完成。
  6. 制定輕量模式慣例：為單檔小修改準備固定的 lightweight inline 指令。
最適合的定位 將它視為 Claude Code 的「工程治理層」，而不是每一個任務都必須完整啟動的固定流水線。
