# Fable 5 Harness — 給 Claude Code 的營運制度包

這是一套 Claude Code 的「操作制度」檔案，2026-07-06 由一個 Claude Fable 5 session 一次性
建立：把一個較強模型 session 的判斷力，寫成之後每個 session（無論主力跑 Opus 或 Sonnet）
都能逐條執行的規則、rubric 和模板。建立後由另一個 fresh-context agent 對抗審查出 16 條問題
並全數修正。

**這份是對外分享版**：以 Opus 為主力調校（opus 為派工階梯頂端、建議 high effort），
`fable` 改成安裝時可選——安裝精靈會問「你有沒有 Fable 5 使用權限」，有就保留 fable 當頂端、
沒有就整列移除，同一份包通吃兩種朋友。

> **最快上手**：在這個資料夾裡打開 Claude Code，隨便說一句「幫我安裝這個」即可——
> 資料夾根目錄有一個 bootstrap `CLAUDE.md` 會自動載入，Claude 會自己去讀 `install_harness.md`
> 並開始。若沒自動開始，把這句貼給它：
> `讀取 install_harness.md，照著裡面的步驟安裝`

## 怎麼安裝（給你的朋友）

1. 把整個 `fable5_harness/` 資料夾複製到自己電腦上任意位置。
2. **在該資料夾裡**開一個 Claude Code session（建議用 Opus）。根目錄的 bootstrap `CLAUDE.md`
   會自動載入，通常你只要說「幫我安裝這個」就會開始；若沒有，對它說：

   > 讀取 install_harness.md，照著裡面的步驟安裝

3. Claude 會自己備份既有設定、複製檔案到 `~/.claude/`、把檔案裡預留的 `{…}` 佔位段落填成
   你機器的實況（會先問你幾個問題：成本傾向、主要用途、語言偏好、以及有沒有 Fable 5 權限），
   最後驗證安裝結果。
   包內檔案已預先通用化：規則與範例不綁定任何作業系統或專案類型，就算安裝方偷懶沒填
   佔位，也不會裝出錯誤的環境指令。
4. 裝完**重啟 Claude Code**（自訂 agent 要重啟才會註冊），之後什麼都不用做——
   `~/.claude/CLAUDE.md` 每個 session 自動載入。

## 這套制度在解決什麼問題

原作環境診斷出三個根因（見 `docs/00-harness-diagnosis.md`，適用於幾乎所有 Claude Code 用戶）：

1. **主對話自己下場做粗活。** 整檔讀大檔案、反覆看截圖、批次改檔都發生在主對話裡，
   幾次就把 context 撐爆觸發壓縮，壓縮後模型開始忘記你講過的約束、重複做過的事。
2. **沒有常駐制度。** 沒有 CLAUDE.md 時，每個新 session 都重新摸索專案怎麼跑、怎麼驗證、
   你有什麼偏好；學到的教訓下個 session 就消失。
3. **「完成」是模型自己說了算。** 「程式看起來沒問題」不是證據；小模型尤其容易高報完成度。

## 包內檔案一覽

| 檔案 | 作用 |
|---|---|
| `home-claude/CLAUDE.md` | 全域路由器（約 50 行）：5 條觸發規則（何時去讀哪份文件）＋ 5 條鐵律。刻意短到每個 session 都會完整讀完 |
| `docs/00-harness-diagnosis.md` | 上述三大診斷與修法——整套規則的「為什麼」 |
| `docs/10-model-dispatch.md` | 派工守則：指揮官不下場、派工必帶三件套（目標動機/驗收條件/回報格式）、模型升降級階梯（錯幾次升級、驗證過的模式降級批量做、每個做法最多重試兩輪）、驗證不自驗 |
| `docs/20-judgment-rubrics.md` | 判斷力 rubric：何時升級模型、何時算真的完成、何時該停下問使用者、什麼訊號代表方向錯了該換路、品質底線。每條附一個正例一個反例 |
| `docs/30-delegation-templates.md` | 五種派工 prompt 模板（搜尋/實作/重構/研究/審查），含驗收條件與回報格式填空、統一的失敗條款 |
| `docs/40-maintenance-protocol.md` | 維護協議：模型可以自己改什麼（查證過的事實）、什麼要先問人（任何規則門檻）、教訓寫回哪裡用什麼格式、膨脹到多長要精簡 |
| `docs/50-letter-to-future-sessions.md` | 給未來 session 的信：最可能的三種制度退化方式（路由失靈、貪方便鬆綁規則、事實過期被誤判成規則失效）與預防法 |
| `agents/verifier.md` | 自訂驗收 agent：fresh-context、只認執行證據（跑測試、實駕頁面、read-back），逐條回報 PASS/FAIL（opus，high effort） |
| `agents/implementer.md` | 為 Opus 調校的實作 agent：接規格明確的實作/重構任務，先自己跑過再回報（opus，high effort）。它是「做的人」，仍由 verifier 另外驗收 |
| `agents/hard-solver.md` | 為 Opus 調校的難題求解器：常規嘗試卡關後才派、會權衡多種解法、誠實回報信心與殘餘風險（opus，max effort）。刻意昂貴，少用 |
| `CLAUDE.md`（根目錄） | bootstrap 引導檔：在此資料夾打開 Claude Code 會自動載入，把 Claude 導向 `install_harness.md`。不會被安裝 |
| `install_harness.md` | 給安裝方 Claude Code 讀的安裝與在地化步驟 |

## 裝完之後的日常長什麼樣

- 你照常對話。模型遇到「要派 subagent」「要宣稱完成」「要改制度檔」這類時刻，會被
  CLAUDE.md 的路由表引導去讀對應文件再動手。
- 實質性的工作完成後，模型會派一個沒看過過程的 `verifier` agent 拿驗收條件去實測，
  而不是自己說做完了。
- 你糾正過的事，模型會按維護協議把教訓寫回文件（改規則本身則一定先問你）。

## 這套能做到什麼、做不到什麼（誠實定位）

先講結論，免得你轉述給朋友時把話講過頭：**這是一層流程與紀律，不改模型權重，不會讓
Opus/Sonnet 變聰明。** 它做的是讓模型「穩定發揮在自己能力的高端、少犯低級錯」。

**它會實際改善的（比原本好，而且是真的）：**
- context 不被主對話撐爆 → 少觸發壓縮 → 不會做到一半忘記你講過的約束。
- 「完成」要有執行證據、還要 fresh-context `verifier` 驗收 → 擋掉「看起來沒問題」的高報。
- 判斷題寫成 rubric、有升降級階梯與 max-effort 難題求解器 → 少走冤枉路、卡關換路而非硬撞。

  一句話：**有紀律的 Opus，會穩定贏過沒紀律的 Opus。**

**它做不到的（別對朋友這樣講）：**
- 不會「逼近 Fable 5 的能力」。遇到超出 Opus 推理上限的題，再多鷹架也不會把它變成更大的
  模型——鷹架墊高的是「下限與穩定度」，不是「能力天花板」。
- 補不了**品味與模糊判斷**（視覺方向、文風、兩個都不錯的方案選哪個）——任何流程都補不了。
  所以制度規定：遇到品味題，模型必須做出 2–3 個真正不同的候選、附上你能直接判斷的證據
  （截圖、樣張），由你來選——不准替你默默決定。

**中間那個合理的內核：** 在**可拆解、可驗證**的工作上（規格明確的實作、重構、有標準答案的
研究），一個被好好編排的 Opus，產出**可以逼近「一個更強模型一發到底會產出的東西」**——因為
流程把它會犯、而強模型本來也不會犯的錯都攔掉了。這是「在這類任務上，結果逼近」，不是「能力
逼近」，差別很重要。

> 誠實但書：以上是**從機制推理**的定位，不是實測 benchmark。想用數據說話的話，可以拿同一批
> 任務做「有制度 vs 沒制度」對照，看完成率／回頭修正次數／驗收通過率。

## 出處

原始需求（fable5.txt）：用僅有一次的 Fable 5 session「立制度，不要拿去執行日常任務」，
把判斷力外化成之後每個較弱模型 session 都能沿用的檔案。本包即該次 session 的全部產出。
