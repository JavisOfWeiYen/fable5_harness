制度之前，
制度之後

這套 harness 不改模型權重，不會讓模型變聰明。它改的是模型工作的方式： 墊高的是下限與穩定度，不是能力的天花板。一句話——有紀律的模型，穩定贏過沒紀律的同一個模型。

2026-07-07 · 依據已安裝的 ~/.claude/ 文件、本機 testing/ 對照實驗，與本 session 的實際觀察

序

裝了什麼

安裝的東西很少：一份約五十行的 ~/.claude/CLAUDE.md（路由器：五條觸發規則＋五條鐵律）、 六份制度文件、三個自訂 agent（verifier／implementer／hard-solver）、 兩個每 session 只提醒一次的 hooks。沒有任何要手動呼叫的指令——路由器每個 session 自動載入， 其餘文件只在對應時機被讀。

差異因此不在「多了什麼功能」，而在同一個模型於幾個關鍵時刻，行為被明文改寫了。 以下逐一對照。

因

安裝之前的日常——三個根因

沒裝之前，幾乎每個 Claude Code 用戶都反覆撞上同三件事（完整診斷見 docs/00-harness-diagnosis.md）：

主對話自己下場做粗活

整檔讀大檔案、反覆看截圖、批次改檔全發生在主對話裡。幾次就把 context 撐爆、觸發壓縮； 壓縮之後，模型開始忘記先前講好的約束、重做做過的事。

沒有常駐制度

每個新 session 都從零開始摸索：專案怎麼跑、怎麼驗證、使用者有什麼偏好。 上個 session 學到的教訓，下個 session 就蒸發。

「完成」由模型自己說了算

「程式看起來沒問題」不是證據。模型高報完成度時，發現問題的成本落在使用者身上—— 通常是在下一次打開頁面的時候。

變

安裝之後——六個時刻的改寫

一、session 的開場

無

之前

開場是一張白紙。模型對這台機器、這位使用者、該去哪裡查規矩，一無所知； 遇事憑訓練時的直覺。

有

之後

路由器自動載入。它刻意短到每個 session 都會完整讀完，核心是一張「觸發時機 → 先讀哪份文件」的表：

markdown · ~/.claude/CLAUDE.md（節錄） 
## Routing table — when a trigger fires, Read the file before acting

| Trigger                              | Read                       |
|--------------------------------------|----------------------------|
| About to spawn a subagent            | docs/10-model-dispatch.md  |
| About to say a task is done          | docs/20-judgment-rubrics.md|
| Writing a delegation prompt          | docs/30-delegation-templates.md |
| The user corrected you               | docs/40-maintenance-protocol.md |

環境事實（WSL2 跨檔案系統 I/O 慢、Playwright 已裝、回應用繁體中文）也寫死在裡面—— 不必每次重新發現，也不會誤判「慢」是「當掉」。

二、主對話的角色

無

之前

主對話是工人：親自讀完 2000 行的檔案、親自掃 repo、親自逐檔修改。 context 是消耗品，用完觸發壓縮，然後開始失憶。

有

之後

主對話是指揮官（鐵律一）：粗活派給 subagent，只取回結論與 file:line 參照。 派工必帶三件套，寫進 prompt 而不是留在腦中：

markdown · docs/30-delegation-templates.md（骨架） 
## Goal & why
{一句話目標，加上為什麼——讓 agent 能自行取捨}

## Acceptance criteria          ← 驗收條件跟著任務走
- [ ] {可執行、可判定的檢查，不是「做好」}

## Report format
PASS/FAIL per criterion, evidence attached

獨立的子任務一批同時派出（parallel-dispatch 規則），不逐一排隊。

三、「完成」的定義

無

之前

「我已經實作了草稿保存——程式碼會在每次變更時寫入 localStorage，所以重新整理後草稿會保留。」 沒有任何東西被跑過。這是宣稱，不是結果。

有

之後

「完成」必須是執行產生的證據（鐵律二）：測試真的跑過、頁面真的用瀏覽器開過、寫出的檔案真的讀回來過。 而且實質性工作的驗收由一個沒看過過程的 fresh-context verifier 執行——做的人不自驗。 本報告誕生的這個 session 就是實例：

verifier 回報 · 本 session 實錄（節錄） 
VERDICT: PASS
1. PASS — diff exited 0; cmp confirms byte-identical, 4935 bytes
2. PASS — grep -c '§ 0, item 4' → 1 ; grep -c '§ 0.4' → 0
3. PASS — backups present: …bak-2026-07-07-1452
4. PASS — zero 'INSTALLER:' placeholders remain
5. PASS — all 6 routing-table targets exist on disk

四、判斷題

無

之前

什麼時候該換個做法、什麼時候該停下來問人、什麼時候算方向錯了——全憑當下的感覺。 卡關時最常見的行為是原地硬撞：同一招重試第五次。

有

之後

每個容易搞砸的判斷都有 rubric（docs/20），各附一個正例一個反例； 模型的升降級走明文的階梯（docs/10）：

  • sonnet日常的做與查——先便宜地試
  • opus同一子任務錯兩次即升級，附上兩次失敗軌跡
  • opus · max efforthard-solver：常規嘗試都卡死後的最後手段，刻意昂貴，少用
  • 問使用者每個做法最多重試兩輪；品味題、風險題不硬決

五、紀律的執行方式

無

之前

就算把規則寫進文件，遵守與否仍靠模型自律。而制度文件自己預言了最可能的退化： session 會略讀規則、然後照舊行事。

有

之後

兩個最關鍵的觸發從自律變成機制。hooks 在「第一次派工前」與「第一次要宣告收工前」 機械式注入提醒，每 session 各只打擾一次：

sh · Stop hook（節錄） 
in=$(cat)
[ "$(printf '%s' "$in" | jq -r .stop_hook_active)" = "true" ] && exit 0
sid=$(printf '%s' "$in" | jq -r .session_id)
m="${TMPDIR:-/tmp}/claude-doctrine-stop-$sid"
[ -e "$m" ] && exit 0; touch "$m"   # 標記檔：每 session 只擋一次，不會無限迴圈
printf '%s' '{"decision":"block",
  "reason":"…完成宣告需有執行證據（20-judgment-rubrics § 2）…"}'

寫這份報告的 session 裡，兩個 hook 都真的觸發過——一次在第一次派 subagent 前， 一次在第一次宣告完工前。

六、教訓的去向

無

之前

你糾正過的事，這個 session 記得，下個 session 忘光。同一個坑，每週重踩一次。

有

之後

維護協議（docs/40）規定教訓寫回哪裡、用什麼格式；跨 session 的交接記在 logbook（docs/50 § Handoff）。改「事實」可以自主（需附驗證），改「規則」一定先問你：

markdown · Lessons learned 的固定格式 
- 2026-07-07 · trigger: {發生了什麼，一行}
  · rule: {下次該怎麼做，一行祈使句}
  · evidence: {怎麼知道的，一行}

文件膨脹也有明文門檻（router 超過約 70 行、單篇文件超過約 250 行就要提案精簡）， 制度本身不會無限長胖。

證

一次實測對照——同一顆彈珠台，做兩次

本機 testing/ 裡有一組對照實驗：同一份需求（純 HTML/CSS/JS 彈珠台， 含同一條「完成後要真的驗證」條款），由沒裝 harness 與裝了 harness 的 session 各做一次， 各自留下開發報告。可比的數字如下：

無制度有制度
驗證環境 py-mini-racer——V8 引擎＋假 DOM，沒有瀏覽器 Playwright＋無頭 Chromium，真實瀏覽器
視覺層驗證 0 張截圖；canvas 繪圖被 stub 成 no-op，自承「未實際觀察」 2 張真實渲染截圖，讀回確認（發光 bumper、計分、生命數）
驗證項目6＋8＝14 項（引擎內）11 項（瀏覽器內）
首輪通過率未載明（報告稱全過） 9/11 → 修一行 → 11/11；失敗證明測試真的在測東西
Console 錯誤無從得知（無瀏覽器）實測為零
可重用資產— 瀏覽器驗證管線寫入記憶，之後的 session 直接沿用

最鋒利的差異只有一句話：兩邊都「跑了測試」，但只有一邊真的看見了自己做出來的東西。 無制度那次的「已驗證」，止步於一個沒有畫面的模擬 DOM；有制度那次被「完成＝執行證據」逼著多走三步 （繞過 PEP 668、手動解出四個系統函式庫、建 debug handle），換到的是真瀏覽器裡的行為與像素。

誠實但書：兩份報告都是各自 session 的自述，通過數是自報而非第三方稽核； 且此任務規模小，harness 的派工優勢在這個實驗裡沒有機會展現——展現的是驗證紀律。 另外這是 n=1 的對照，是機制差異的例證，不是統計結論。

界

做不到的事——界線

制度的價值有一半在於它誠實標明自己的邊界：

  • 不會提高能力天花板。遇到超出模型推理上限的題目，再多鷹架也不會把它變成更大的模型。 可以期待的是：在可拆解、可驗證的工作上，被好好編排的模型，結果逼近一個更強模型一發到底的產出—— 因為流程把它會犯、而強模型本來就不犯的錯攔掉了。「結果逼近」不是「能力逼近」。
  • 補不了品味。視覺方向、文風、兩個都不錯選哪個——制度的解法不是替你決定， 而是規定模型必須做出二到三個真正不同的候選、附上可直接判斷的證據（截圖、樣張），交給你選（鐵律四）。
  • 有代價。派工與 fresh-context 驗收都消耗 token 與時間；對一望即知的小修小改， 制度也明文允許自檢（inline edit 的例外條款），不是每件事都套全流程。

跋

收尾

裝與不裝之間，模型沒有變聰明一分。變的是：context 不再被粗活撐爆、 「完成」兩個字重新有了重量、判斷題有了成文的答案、你糾正過的事會留下來。

不完美的模型，配上承認不完美的制度——這大概就是這套東西最侘寂的地方。
