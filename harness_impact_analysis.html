Fable 5 Harness · Impact Analysis

制度之前，
制度之後

這套 harness 不改模型權重，也不把 Opus 變成 Fable。它改的是模型工作的姿勢：少把 context 燒在粗活上，少用「看起來對」冒充完成，讓每個重要判斷都有可查的規則。

一句話

有紀律的模型，穩定贏過沒紀律的同一個模型。提升的是下限與可驗證性，不是能力天花板。

侘寂版單頁報告語法高亮本機對照觀察
序

裝了什麼

安裝的東西很少：一份短 router、六份制度文件、三個自訂 agent、兩個選配 hooks。沒有新產品介面；改變發生在 Claude Code session 每次要派工、驗收、卡關、收尾的時刻。

因

安裝之前的三個根因

沒裝之前，問題不是模型完全不能做事，而是它在幾個固定位置會用錯工作方式。

01

主對話自己下場做粗活

整檔讀大檔、反覆看截圖、批次改檔都塞進主對話。context 撐爆後觸發壓縮，模型開始忘記前面講好的限制。

02

沒有常駐制度

每個新 session 都重新摸索怎麼跑、怎麼驗、你偏好什麼。學到的教訓沒有固定去處。

03

完成由模型自己說了算

「程式看起來沒問題」不是證據。沒跑測試、沒打開頁面、沒讀回檔案，就只是宣稱。

變

六個時刻被改寫

harness 的價值不在新功能，而在把容易失手的時刻變成有規則、有證據、有交接的流程。

01 · 開場

無

session 是白紙。模型靠直覺猜這台機器、這個使用者、這個 repo 的工作方式。

有

根 router 自動載入：觸發派工、驗收、維護、複雜任務時，先讀對應制度文件。

02 · 主對話

無

主對話是工人：親自掃 repo、親自讀長檔、親自跑迭代。context 被拿去當工作台。

有

主對話是指揮官：粗活派給 subagent，回來只收結論、證據與 file:line。

03 · 完成

無

「我實作了，所以應該會動。」沒有瀏覽器、沒有測試、沒有 read-back。

有

完成必須有執行證據；實質工作由 fresh-context verifier 驗收，做的人不自驗。

04 · 卡關

無

同一招重試第五次，越修越大，卻說不清楚為什麼這一步還服務目標。

有

升級與換路都有條件：同一子任務錯兩次升級；第三輪必須換模型、換拆法或問使用者。

  • sonnet日常明確任務。
  • opus同一子任務失敗兩次，附完整失敗軌跡。
  • hard-solver常規嘗試卡死後的 max effort 最後手段。
  • user品味題、破壞性操作、真正有歧義的選擇。

05 · 紀律

無

規則寫在文件裡，但模型略讀後照舊。制度靠自律，最容易退化。

有

once-per-session hooks 在第一次派工前注入提醒；做過事（派過工或改過檔）的 session 第一次收工前注入 done-check——把最高價值觸發變成機制，純問答 session 零成本。

06 · 記憶

無

你糾正過的事，只活在這個 session。下個 session 又踩一次。

有

教訓有去處：專案事實進 memory，跨專案 doctrine 進 docs，handoff 進 logbook；改規則先問人。

證

一次實測對照

同一份 Prompt Pinball 需求，沒裝與裝了 harness 各做一次。這不是統計結論，而是機制差異的切片。

觀察點無制度有制度
驗證環境V8 引擎 + 假 DOM，沒有真瀏覽器Playwright + 無頭 Chromium，真頁面
視覺驗證0 張截圖，canvas 被 stub 成 no-op2 張真實截圖，確認 bumper 發光、計分、生命數
首輪通過未載明，報告稱全過9/11 → 修一行 → 11/11；失敗證明測試真的在測
Console無從得知實測為零錯誤
可重用資產無瀏覽器驗證管線寫回記憶，後續 session 可沿用

最鋒利的差異只有一句話：兩邊都「跑了測試」，但只有一邊真的看見了自己做出來的東西。

界

做不到的事

制度的價值有一半在於承認邊界。它不假裝神奇，也不替使用者做品味決定。

  • 不提高能力天花板。遇到超出模型推理上限的題，再多流程也不會把它變成更大的模型。
  • 不補品味。視覺方向、文風、兩個都不錯選哪個，制度要求模型提出 2–3 個候選並交給你選。
  • 有成本。派工與 verifier 都消耗 token 和時間；很小的 inline edit 仍可自檢。
圖

檔案地圖

安裝後真正生效的是 ~/.claude/ 內的 payload；repo 根目錄只是 installer package。

  • CLAUDE.md全域 router：觸發時先讀哪份文件。
  • docs/10-model-dispatch.md模型、派工、升降級、驗證契約。
  • docs/20-judgment-rubrics.md完成、升級、停下問人、方向錯誤。
  • docs/30-delegation-templates.md五種常見任務的 prompt 骨架。
  • docs/40-maintenance-protocol.md制度如何被安全更新。
  • docs/50-letter-to-future-sessions.md退化模式與 handoff log。
  • agents/verifier.mdfresh-context 驗收者。
  • hooks.json選配 once-per-session hook 範本，只能 merge。
跋

收尾

裝與不裝之間，模型沒有變聰明一分。變的是：context 不再被粗活撐爆，「完成」重新有了重量，判斷題有了成文的答案，你糾正過的事會留下來。

不完美的模型，配上承認不完美的制度——這大概就是這套東西最侘寂的地方。
