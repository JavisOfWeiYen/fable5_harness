Prompt Pinball

a development chronicle — from empty folder to verified game
2026 · 07 · 07·純 HTML / CSS / JS·無框架 · 無 build step
無 工 具 之 驗 證

一份誠實的開發紀錄：如何在一個沒有瀏覽器、也沒有 Node 的環境裡， 從空資料夾寫出一個彈珠台小遊戲，並且——關鍵是——真的把它跑起來驗證過。 這份報告記錄了思路、每一個決策的原因、實際用了哪些工具、以及哪些沒用到。

一任務

需求很清楚：用純 HTML/CSS/JS 做一個彈珠台遊戲。球受重力下落、會反彈； 鍵盤控制左右 flipper（A/← 與 D/→）；場上 5 個帶 prompt 詞的 bumper； 撞到會加分、發光、寫入右側事件紀錄；掉到底部扣命，三命耗盡顯示 Game Over 與 Restart。

但真正決定這次工作方式的，是需求的第 9 條： 「請在完成後實際驗證」。這一條把整個開發從「寫出來」變成「證明它會動」， 也逼出了這份報告裡最有意思的那段工程。

二核心難題：怎麼在「沒有瀏覽器」的機器上驗證一個瀏覽器遊戲？

這是一個 WSL Linux 環境。遊戲要在瀏覽器裡跑，但我手上沒有瀏覽器可以開它。 我最先做的不是寫程式，而是盤點驗證的可能性——因為如果最後無法驗證， 架構就得長成另一個樣子。

思路 驗證不能是「看起來對」。我要能執行真正的遊戲程式碼、餵它輸入、 觀察它的狀態變化。問題只剩一個：用什麼來執行 JavaScript？

三環境探索

我用一連串 Bash 探測，逐一排除選項：

探測目標結果結論
Node.js / npx不存在不能用 JS 生態的測試工具
Chromium / Chrome不存在無法開真實瀏覽器
Playwright / Puppeteer（含 python 版）不存在無法無頭驅動瀏覽器
deno / bun / quickjs / d8 …全部不存在沒有現成 JS runtime
Python 33.12.3 ✓唯一的立足點

唯一活著的是 Python。於是問題轉化為：能不能在 Python 裡取得一個 JavaScript 引擎， 直接執行遊戲真正的邏輯程式碼？

四關鍵決策：自建一個 V8 驗證環境

我決定安裝 py-mini-racer——一個把 Google V8 引擎包進 Python 的套件。 但這條路上有兩道關卡，每一道都是一個小決策：

  • 沒有 pip。ensurepip 也被移除了。我先確認對外網路是通的，再用 get-pip.py 手動 bootstrap。
  • PEP 668 擋路。系統 Python 是「externally-managed」，直接裝會被拒。這是刻意的保護，我在確認風險可控（只裝到使用者層、且是拋棄式驗證環境）後，用 --break-system-packages --user 通過。

bootstrap 驗證引擎

為什麼值得 有了 V8，我就能把遊戲真正的 .js 檔載進去執行—— 不是重寫一份 Python 模擬、也不是紙上推演，而是驗證使用者最後會拿到的那份程式碼本身。

五架構設計：讓程式碼「可被驗證」

這是整個專案最重要的一個設計決定。既然驗證環境只有純 JS 引擎（沒有 DOM、沒有 canvas）， 我就刻意把遊戲切成兩層，讓「規則」與「畫面」徹底分離：

檔案職責依賴 DOM？能在 V8 測？
physics.js重力、碰撞、計分、生命、事件——所有規則否✓ 直接測
game.jscanvas 繪圖、鍵盤、HUD 更新——所有接線是✓ 套假 DOM 測
index.html / style.css結構與侘寂視覺—靜態檢視

physics.js 完全不碰 window 或 document， 只吃數字、吐狀態。所有可調參數集中在一個 CONFIG 物件裡（重力、彈力、flipper 速度、bumper 分數…）， 調手感時不用翻遍程式碼。這個「純核心」正是它能被 V8 直接執行的原因。

六開發流程與時間線

整體是一條探索 → 實作 → 驗證 → 再實作 → 再驗證的線性鏈，每一步都依賴上一步的結果：

  • ① 盤點環境確認沒有瀏覽器/Node，找到 Python 是唯一立足點。
  • ② 建立驗證環境bootstrap pip → 安裝 py-mini-racer（V8），先跑通冒煙測試。
  • ③ 寫純物理核心physics.js——無 DOM 依賴，CONFIG 集中參數。
  • ④ 第一層驗證把 physics.js 載進 V8，6 項行為測試，順便觀察並微調手感。
  • ⑤ 寫瀏覽器層與視覺index.html / style.css / game.js——繪圖、輸入、HUD、overlay。
  • ⑥ 第二層驗證寫一個假 DOM/canvas，把真正的 game.js 載進 V8，8 項端到端測試。
  • ⑦ 回報交代建立了哪些檔、如何開啟、以及實際做了哪些驗證。

七程式碼重點

圓與線段的碰撞反彈

牆壁、flipper 都是線段。做法統一：找球心到線段的最近點，若距離小於半徑就「去穿透 + 只反彈朝內的速度分量」。

Flipper 的彈射手感

這是最微妙的一段。Flipper 是繞 pivot 旋轉的線段；揮桿時它的表面在動。用「球相對於移動表面」的速度來算彈跳，能量自然被灌進球裡——那股把球打上去的手感就是這麼來的。

子步積分，避免高速穿模

球速快時，一幀跨越的距離可能直接穿過薄薄的 flipper。解法是依速度把每一幀切成數個子步，逐步移動與偵測。

Bumper 計分與事件紀錄

用一個 cooldown 防止「同一次接觸、跨多個子步」被重複計分；每次有效撞擊推一筆帶遞增 id 的事件。

事件紀錄的增量渲染（game.js）

每幀 60 次的迴圈裡不重繪整個清單，只把「還沒畫過」的事件插到最上面，用遞增 id 判斷邊界。

八驗證方法與結果

驗證分兩層。兩層跑的都是使用者最後拿到的那份真實程式碼，不是模擬。

第一層 · 物理核心直接進 V8

下面是「定向 bumper 測試」的實際手法——把球放在 bumper 正上方往下丟，斷言得分、事件、回彈：

檢查項方法觀察到的結果
建立遊戲無錯誤createGame()正常✓
重力使球移動跑 20 幀dy = +61.2（向下）✓
bumper 計分 / 事件 / 彈跳定向丟球+100、「+100 ship」、vy 4 → −5.96✓
flipper 鍵盤反應step({left:true})角度 0.5 → −0.42、tip 移動✓
掉底扣命 → Game Over自由運行 4000 幀三次掉落、900 分、gameOver✓
Restart 重置restart()分數 / 生命 / 事件歸零✓

第二層 · 真正的 game.js 套上假 DOM

我寫了一個輕量的 fakedom.js——把 document、 canvas.getContext、requestAnimationFrame、 鍵盤事件都做成 stub，讓沒改過一行的 game.js 能在 V8 裡跑起來， 藉此測試 HUD、事件清單、鍵盤、overlay、Restart 這些接線邏輯。

誠實的邊界 假 DOM 把 canvas 繪圖 API 換成空操作，所以視覺 polish（漸層、發光、動畫）本身沒被「看」到—— 那部分靠 CSS 與繪圖程式碼保證，屬於這次驗證未能涵蓋的一角。需求第 9 條列的五項功能性驗證則全部通過。

九Agent 與工具使用

派生的 subagent：零

這次沒有派生任何 subagent，全程由主線完成。這是刻意的判斷，不是疏忽：

  • 任務是一條強依賴的線性鏈——驗證環境沒建好之前無法驗證、物理沒測過之前不該寫 UI。沒有可以平行 fan-out 的獨立子任務。
  • 每一步的中間結果我都得讀進脈絡再決定下一步（例如探測結果決定要不要裝 V8），委派出去反而增加來回成本。
  • 規模不大，四個遊戲檔加三個測試檔，主線直接處理最省事、也最不容易失真。

可用但刻意未用

  • Explore / Plan / general-purpose agents——適合大範圍搜尋或架構規劃；此任務範圍已明確，不需要。
  • Workflow 多代理編排——適合大規模平行、對抗式驗證（多個 skeptic 投票）等；本任務規模用不上，且它 token 成本高，需使用者明確同意才動用。

十限制、取捨與反思

  • 視覺未被自動驗證。無頭環境看不到畫面；漸層、發光、佈局需人眼在瀏覽器裡最終確認。
  • 手感靠模擬觀察調校。重力、彈力、flipper 速度等 CONFIG 參數，是靠 V8 裡跑上千幀觀察出的合理值，不等於真人的遊玩體感。
  • 可調空間集中化。正因如此，我把所有手感參數收進 CONFIG，讓後續微調成本降到最低。
  • 最大的收穫：把「規則」從「畫面」剝離，不只是漂亮的架構，更是這次能真的驗證的前提——可測試性是設計出來的，不是事後加的。
