序

三條寫在牆上的心法

在動手之前,有三條原則決定了這次每一個選擇。它們來自全域 doctrine,不是臨時想出來的:

「完成」的定義是執行過的證據——一次跑過的測試、一次真實的瀏覽器操作、一張渲染出來的截圖。
「程式碼看起來對」不算完成。
  • 驗證優先。 需求第 9 條白紙黑字要求「實際驗證」。所以我做的第一個判斷不是「怎麼寫遊戲」,而是「這台機器要怎麼才能真的打開頁面看它跑」。
  • 不要陷進雜草裡。 大量檔案掃描、逐行閱讀應該交給 subagent,主線只收結論。
  • 品味交給使用者,工程自己扛。 視覺風格會給選項;但「能不能跑」這種事實問題,我得自己用證據回答。
01

起點:一次熱身,一個大任務

對話從一個熱身開始——你要我測 verifier agent 能不能 spawn、hooks 能不能觸發。這其實替後面鋪了路:我因此先摸清楚了這個 harness 能派 subagent、settings 裡目前沒有任何 hook。

接著才是主菜:用純 HTML/CSS/JS、不用框架、不用 build step,做一個叫 Prompt Pinball 的彈珠台小遊戲——重力落球、牆與 bumper 反彈、鍵盤控制左右 flipper、5 個帶 prompt 詞的 bumper、計分、生命、事件紀錄、Game Over 與 Restart,而且要有 polish。

當下的判斷 · 這是一個「規格清楚的建構任務」。doctrine 會建議派 implementer 去做。但真正的難點不在寫遊戲——而在驗證,因為我還不知道這台機器能不能跑 JS。所以我把力氣先花在打通驗證管道。

02

環境探勘:先確認手上有什麼

與其假設,不如探測。我一次列出所有可能的 JS runtime 與瀏覽器,結果幾乎全軍覆沒:

找什麼結果意義
node / deno / bun全都沒有不能直接跑 JS 驗證邏輯
chromium / chrome沒有沒有現成瀏覽器
playwright / selenium沒有沒有自動化管道
python3 + pip3有(3.12)這就是唯一的立足點

結論很清楚:Python 是唯一的槓桿。那就用它把一個真的瀏覽器架起來——Playwright + headless Chromium,而且要裝在原生 ext4 那側(快),不要碰慢的 /mnt/c 9p 掛載。

03

打通驗證管道:一場免 root 的修補

這是整個任務最像「工程」的一段。它不是一路順風,而是連撞三道牆、再逐一補起來——很有侘寂的味道:承認裂縫,然後用金繕把它接好。

裂縫一 · PEP 668 擋下系統安裝

直接 pip3 install 被「externally-managed」擋下。想用 venv,卻發現 ensurepip 缺失,venv 生不出 pip。解法:退一步用 user-site 安裝。

裂縫二 · Chromium 缺 4 個系統 lib,而且沒有 root

Chromium 下載好了,一啟動就 TargetClosedError。ldd 一看,缺 libnss3 / libnspr4 / libnssutil3 / libasound。裝它們要 apt/root——但這台沒有免密碼 sudo。

金繕 · 不靠 root 也能補

apt-get download 不需要 root,可以只把 .deb 抓下來;再用 dpkg-deb -x 解壓到家目錄,最後用 LD_LIBRARY_PATH 讓 Chromium 找到這些 lib。

接著一個最小 smoke test:啟動 → 設一段 HTML → 讀回文字。印出 CHROMIUM_LAUNCH_OK 的那一刻,驗證管道就通了。

一個判斷 · 我其實可以直接請你用 ! sudo … 裝 lib。但 doctrine 要我先窮盡自主解法;免 root 這條路可行,就不打擾你。若當時走不通,才會回頭請你出手。
04

遊戲架構:三個檔案、一套物理

管道通了,才開始寫遊戲。刻意分成三個純檔案,直接用瀏覽器打開 file:// 就能跑:

檔案職責
index.html版面骨架:canvas、分數、生命、事件紀錄、Restart、Game Over overlay
styles.css深色一致配色、發光/動畫、grid 布局(純 CSS)
game.js物理引擎:重力、碰撞解算、bumper、flipper、計分/生命/紀錄

一套解算,重複使用

牆、漏斗導軌、flipper 全都是「線段」。所以我只寫一份球 vs 線段的碰撞解算:找最近點 → 推出接觸面 → 沿法線反射(帶恢復係數)。省下大量重複程式,也讓行為一致。

flipper 為什麼能「彈射」球

flipper 是一根繞著軸旋轉的線段。撥動時它有角速度 ω,接觸點的表面速度就是 ω × r。把這個速度餵進上面的解算,球就會被真正「打」出去,而不只是靜態反彈。

其餘細節:用子步進(把每幀切成小步)避免高速穿透;每個 bumper 有冷卻時間,避免一次接觸連續灌分。

05

把「可測性」寫進設計裡

這是整個做法的關鍵一步,也是我最想強調的:我不靠隨機物理去碰運氣驗證,而是把遊戲內部掛到 window 上,讓測試能確定性地驅動它。

為什麼重要 · 這些方法呼叫的是遊戲真正的程式路徑(同一個計分函式、同一個物理步進)。所以測試既是真實的、又是可重現的——不會因為球剛好沒撞到而 flaky。
06

驗證流程,與那次很有教育意義的失敗

我用 Playwright 寫了 11 項驗收,對應你需求裡列的每一條。第一次跑:9 / 11。兩個 flipper 測試掛了——角度完全沒變。

診斷:我在測試前呼叫了 pause() 想「隔離輸入」,但 flipper 角度正是在物理步進裡才更新的;暫停之後,按鍵設了 pressed=true,角度卻永遠不會被積分。這是測試的錯,不是遊戲的 bug——真實遊戲裡 loop 在跑,按鍵完全有效。

07

用到的 agent、工具與流程

Agent

  • verifier agent —— 在熱身階段依你要求 spawn 過一次,確認 harness 能派 subagent、能繼承工作目錄、能執行 shell(同步模式,約 7.5 秒完成)。
  • 主線自建 —— 遊戲本身我在主對話裡完成,沒有再派 implementer/verifier subagent。原因是:任務範圍聚焦(建立三個新檔),而且我用 headless 瀏覽器執行過的證據直接滿足了「done = executed evidence」這條不變量——驗證是靠真實瀏覽器,不是靠「讀程式覺得對」。

08

整條開發流程

  1. 熱身 · 測 harness 能力
    spawn verifier、確認 hooks 現況。順手摸清這台能做什麼。
  2. 判斷 · 難點在驗證,不在遊戲
    需求要求實際驗證 → 先解決「怎麼真的打開頁面」。
  3. 探勘 · 列出所有 runtime/瀏覽器
    全都沒有,只剩 python3 + pip3。確立唯一槓桿。
  4. 修補 · 免 root 架起 headless Chromium
    PEP 668 → user-site;缺 lib → apt-get download + dpkg-deb + LD_LIBRARY_PATH。smoke test 通過。
  5. 建構 · 三檔遊戲 + 可測性設計
    共用線段碰撞、flipper 角速度彈射、子步進防穿透;把 debug handle 掛上 window。
  6. 驗證 · 11 項、9/11 失敗、診斷、修好
    pause 凍住 flipper 積分 → 改 resume → 11/11 全綠 + 截圖。
  7. 收尾 · 存記憶、回報
    把可複用的環境能力寫進 memory,誠實回報做了哪些驗證。
09

反思與可複用資產

  • 最大的價值不是遊戲,是那條驗證管道。 全域筆記原本寫「沒裝 Linux 瀏覽器 driver」——現在這句已經不成立。我把安裝步驟與 LD_LIBRARY_PATH 的坑寫進 memory,下次任何 web/UI 任務都能直接用,不必再花十分鐘重踩。
  • 可測性要在寫程式時就設計,不是事後硬湊。 window.__PINBALL__ 讓驗證從「祈禱球會撞到」變成「確定性地擺球、推進、斷言」。
  • 那次失敗是好事。 它證明測試真的在測東西——一個會因為我自己設錯 pause 而抓到問題的測試,才是有效的測試。
  • 誠實邊界。 我沒有假裝派了 verifier subagent 去認證遊戲;我如實說明是主線自建 + 瀏覽器執行證據。判斷也如實交代:免 root 走得通,才沒去打擾你要 sudo。

侘寂教人欣賞不完美與修補的痕跡。這次的裂縫——被擋下的安裝、缺失的 lib、掛掉的測試——沒有被藏起來,而是一道道用金繕接好,最後成了比「一開始就順」更值得記錄的東西。
