# Apex Sprint Challenge

一款俯視角（top-down）F1 風格開輪式賽車遊戲，完全以原生 **HTML / CSS / JavaScript / Canvas**
實作，不使用任何框架、遊戲引擎、外部函式庫或建置流程。玩家將在虛構賽道
「**Solvik Grand Loop**」上與三名 AI 對手進行三圈短程競賽，需管理煞車點、
輪胎、ERS、DRS 與進站策略。

所有車隊、車手、賽道皆為虛構，不含任何真實 F1 隊伍或商標。

---

## 1. 執行方式

**直接用瀏覽器開啟 `index.html` 即可遊玩**（雙擊或用瀏覽器開啟本機檔案）。

若你的瀏覽器對 `file://` 有額外安全限制（例如某些瀏覽器對 `<canvas>` 或
`localStorage` 在 `file://` 下的行為較保守），可在本資料夾下啟動一個簡單的
本機伺服器後改用 `http://localhost:8000` 開啟：

```bash
python -m http.server 8000
```

不需要 `npm install`、不需要任何建置流程。

| --- | --- |

| `S` / `↓` | 煞車 |

當焦點停留在輸入框、下拉選單等可輸入元件時，上述快捷鍵會被忽略，不會誤觸。

## 3. 檔案結構

```
index.html    頁面骨架、Canvas、HUD 與選單 DOM
styles.css    版面與視覺樣式
storage.js    localStorage 存取（防毀損）與遙測 CSV 匯出
physics.js    車輛物理：油門/煞車/阻力/側向抓地與打滑/輪胎/油量/ERS/碰撞
track.js      賽道幾何：中心線、檢查點、DRS 區、Pit Lane、表面判定
ai.js         AI 車手：追點轉向、依彎道曲率煞車、ERS/DRS/進站決策
race.js       比賽規則：起跑燈、圈數/檢查點驗證、排名、DRS 資格、Pit 機制、暫停/重置、遙測
game.js       主迴圈、輸入處理、Canvas 繪圖、HUD、選單、Demo Race
tests.html    瀏覽器測試頁面（載入正式模組並顯示結果）
tests.js      14 項自動化測試，直接呼叫上述正式函式
README.md     本文件
```

所有 `.js` 檔皆以一般 `<script>` 標籤載入（非 ES Module），每個檔案在
`(function(global){ ... })(window)` 內把自己的公開 API 掛在
`window.Physics` / `window.Track` / `window.Race` / `window.AI` /
`window.GameStorage` 之下，避免污染全域變數，同時可直接以 `file://` 開啟。

## 4. 車輛物理設計（physics.js）

物理以**固定時間步進 60Hz**（`Physics.FIXED_DT = 1/60`）更新，由 `game.js`
的主迴圈以「累積器（accumulator）」模式呼叫，每畫面最多執行
`Physics.MAX_STEPS_PER_FRAME`（8）次子步進；若瀏覽器卡頓造成單幀時間過長，
多餘時間會被捨棄而不是一次補完，因此車輛不會「瞬間穿越賽道」。

車輛狀態以車身座標系的 `vForward`（前後速度）與 `vLateral`（側滑速度）表示：

- **油門/煞車**：依引擎推力/煞車力除以總質量（乾重 + 油量）換算加速度。
- **空氣阻力與滾動阻力**：阻力隨速度平方增加，DRS 啟用時降低阻力係數；
  草地上滾動阻力與阻力係數皆提高。
- **轉向與速度相關性**：轉向增益在低速時較弱、在參考速度附近最強、
  超過參考速度後略為下降，模擬「車速越高轉向越不敏感」。
- **側向抓地力與打滑**：以目前轉向角速度與前進速度估算所需向心加速度，
  若超過輪胎/路面所能提供的最大側向加速度即進入打滑狀態，產生側滑速度，
  並隨抓地力逐漸衰減回正。草地抓地力係數明顯較低。
- **牆壁與車對車碰撞**：`resolveWallCollision` / `resolveCarCollision` 在偵測到
  幾何重疊時，同時做「位置修正」（推出重疊量）與「速度修正」（沿法線方向
  反彈+阻尼），確保車輛不會卡在牆內或彼此永久重疊；油門在任何速度下都持續
  有效，因此完全靜止後仍可重新起步。

## 5. 賽道與檢查點判定（track.js / race.js）

賽道中心線由一組手動放置的路點，以 **Catmull-Rom 雲形線**平滑後，再依
弧長重新取樣為等間距的樣本陣列（約每 4 個世界單位一個樣本），因此賽道是
一個保證封閉、無縫的迴圈，包含：

- 一條長直線（起跑直線 + Pit Lane 平行段）
- 一個大半徑高速彎
- 一個髮夾彎（樣本點密集，曲率大）
- 一組連續彎（esses）
- Pit Lane、Pit Box、起跑線、3 個依序檢查點、DRS Zone 與 Detection Point

車輛在世界座標的位置，透過 `Track.getProgress(x, y)` 反查到「沿賽道弧長距離
（distanceAlong）」與「垂直於中心線的側向偏移（lateralOffset）」，此距離值
是圈數/檢查點判定的唯一依據，**不是單純比對是否碰到終點線像素**：

1. 每輛車維護 `nextCheckpointIndex`（0~3），僅會依序點亮，不能跳過。
2. 偵測到「正向穿越終點線」（distanceAlong 由接近賽道總長跳回接近 0）時，
   只有在 `nextCheckpointIndex` 已達 3（三個檢查點皆已依序通過）才會讓
   `lap += 1` 並重置檢查點狀態；否則這次穿越不計入圈數。
3. 偵測到「反向穿越終點線」（distanceAlong 由接近 0 跳到接近賽道總長）
   永遠不會增加或減少圈數，避免倒車洗圈數。

排名（`Race.computeStandings`）以 `lap * 賽道總長 + distanceAlong` 的
連續進度值排序，而非單純比較 `lap` 整數，因此同一圈內誰跑得比較遠會被
正確反映在名次上。

## 6. AI 設計（ai.js）

AI 與玩家共用完全相同的物理與規則路徑，**輸出的是與玩家相同格式的輸入**
（`{throttle, brake, steer, ers, drsRequest}`），不會用固定速度沿路徑瞬移：

- **追點轉向**：依目前速度動態調整前視距離（look-ahead），在賽道中心線上
  取一個前方目標點（可含每輛車略為不同的側向偏移，避免完全疊線），以
  pure-pursuit 方式算出轉向量。
- **依彎道曲率煞車/加速**：在前視距離內取樣賽道曲率最大值，反推「安全過彎
  速度」（`v = sqrt(可用側向加速度 / 曲率)`），目前車速超出安全速度太多就煞車，
  否則維持油門；Hard 難度的抓地安全係數更高、煞車容忍誤差更大，因而
  **煞車較晚、過彎速度較高**。
- **脫離賽道恢復**：目標點恆在賽道中心線上，AI 打滑或衝出賽道後轉向會自然
  把車導回賽道；朝向與目標點夾角超過 90 度時視為「面向錯誤方向」，改為
  緩煞車而非全油門硬衝。
- **ERS 使用**：在曲率較低（接近直線）且電量高於難度門檻時啟用，Hard 難度
  門檻更低、用得更積極。
- **DRS 使用**：直接呼叫 `Race.checkDrsEligibility`（與玩家按 F 時檢查的
  函式完全相同），符合條件即要求開啟，不會另外實作一套規則。
- **進站決策**：輪胎磨耗超過難度門檻（Normal 62% / Hard 80%）且已完成至少
  一圈時，設定 `car.pit.wantsToPit`；接近自己被分配到的 Pit Box
  （依車輛索引固定分配）時會覆寫目標點與速度上限，導引車輛平順減速停入
  Pit Box，完成換胎後自動恢復正常賽道目標並重新加速離開。

提供 **Normal** 與 **Hard** 兩種難度（主選單選擇，套用於全部三輛 AI）。

## 7. 輪胎、油量、ERS、DRS 與進站規則

- **輪胎**：具備胎溫（`tempC`）、磨耗（`wearPct`）與抓地係數（`gripFactor`）。
  胎溫低於理想區間或高於理想區間都會降低抓地力，磨耗越高抓地力越低；
  打滑與重煞車會加速升溫與磨耗，草地行駛則會讓胎溫額外降溫。
- **油量**：隨油門使用量持續下降，不會變成負值；總質量隨油量降低而減少，
  加速度（力 / 質量）因此隨之些微改善。
- **ERS**：按住 `Shift` 且電量 > 0 時提供額外加速度，電量持續下降；煞車時
  依煞車力道回充；電量永遠鎖在 `[0, 100]` 之間，耗盡後即使按住也不再有
  額外推力。
- **DRS**：只有在 `Track.isInDrsZone` 為真、車輛已完成至少一圈（第一圈禁用）、
  且在該圈的 Detection Point 與前車距離夠近（以距離差換算成時間差，
  ≤ 1 秒視為合格）時才能啟用；煞車或離開 DRS 區域會自動關閉；玩家按 `F`
  但不符合條件時，畫面會顯示明確原因（例如「DRS unavailable on lap 1」）。
- **Pit Stop**：Pit Lane 全程有速度限制，超速會顯示警告並持續累加時間罰則；
  玩家必須實際把車停入指定 Pit Box（位置 + 速度皆需在容許範圍內）並靜止
  超過 2.5 秒才會觸發換胎（磨耗歸零、胎溫重置為較低的起始溫度）；正常賽道
  上沒有任何「一鍵換胎」的捷徑。

## 8. Demo Race（自動展示模式）

主選單的「Demo Race」會建立一個 `mode: 'demo'` 的比賽狀態，**四輛車（含原本
玩家位置）全部由 `AI.computeInput` 控制**，每一輛車仍然逐幀呼叫與正常比賽
完全相同的 `Race.update`／物理／規則路徑 —— 沒有任何預先寫死的名次或直接
竄改座標的「假比賽」。Demo Race 可以隨時暫停、繼續與重新開始，結束後一樣會
顯示完整的最終成績畫面。`tests.js` 的第 14 項測試會逐幀檢查每輛車單一 tick
的位移量，確保不存在「瞬移」情形。

## 9. 測試方式（tests.html / tests.js）

用瀏覽器打開 `tests.html`（或 `http://localhost:8000/tests.html`），頁面載入
後會自動：

1. 依序載入 `storage.js / physics.js / track.js / race.js / ai.js`（正式模組）
   與 `tests.js`。
2. 執行全部 14 項測試並即時顯示每項的名稱、PASS/FAIL、實際錯誤訊息，以及
   總通過/失敗數。

測試涵蓋（皆直接呼叫正式模組的函式，未另外重寫規則）：

1. 未依序通過檢查點時不可增加圈數
2. 依序通過檢查點並穿越終點線時圈數 +1
3. 倒車重複穿越終點線不能增加圈數
4. 暫停後比賽時間與車輛位置不再更新
5. ERS 不會低於 0 或高於 100
6. ERS 耗盡後不再提供額外動力
7. 第一圈禁止使用 DRS
8. 離開 DRS Zone 後自動關閉
9. 輪胎磨耗增加時抓地力下降
10. Pit Stop 完成後輪胎磨耗歸零
11. Reset 能完整恢復初始比賽狀態
12. 排名計算會考慮賽道內進度而非只比較圈數
13. 損壞的 localStorage 資料不會讓遊戲崩潰
14. Demo Race 使用真實更新流程而非預先寫死結果（含逐幀位移合理性檢查）

## 10. 遙測 CSV 格式

比賽（非 Demo Race）進行中，每 200ms 記錄一次玩家車輛的狀態，按下結算畫面
的「Export Telemetry CSV」即可下載 `apex-sprint-telemetry.csv`，欄位如下：

```
timeMs, lap, x, y, speed, throttle, brake, steer,
tireTempC, tireWearPct, ersPct, drsActive, onTrack
```

- `timeMs`：比賽時鐘（毫秒，暫停時不累加）
- `lap`：當時所在圈數（1-based）
- `x, y`：世界座標
- `speed`：合成速度（世界單位/秒）
- `throttle, brake, steer`：當下輸入值
- `tireTempC, tireWearPct, ersPct`：輪胎溫度/磨耗百分比、ERS 電量百分比
- `drsActive, onTrack`：布林值以 `1`/`0` 表示

## 11. 已知限制

- 物理模型為「可理解、一致」的 arcade 風格模擬，並非精確的多體/輪胎力學模型。
- DRS 資格判定採用「距離差 / 目前車速」換算的近似時間差，未精確模擬真實
  FIA 規章的判定方式（規格書允許此簡化）。
- Pit Box 位置固定依車輛索引分配，不會動態調度或避讓其他車輛。
- 賽道、車隊、車手皆為虛構設計，未嘗試還原任何真實賽道座標。
- HUD／選單使用 DOM 元素疊加於 Canvas 之上，而非全部繪製在 Canvas 內。
- 未實作音效（`GameStorage` 已預留 `masterVolume` / `sfxVolume` 設定欄位，
  但目前沒有音訊播放邏輯）。

## 12. 尚未完成 / 無法在本開發環境驗證的項目

本遊戲的核心邏輯（物理、賽道幾何、圈數/檢查點規則、ERS/DRS/輪胎/Pit
Stop、AI、Demo Race、排名、localStorage 容錯）已透過將 Node.js 18
（於本機以無 root 權限手動解壓 `.deb` 套件取得，非透過 npm 安裝）搭配
DOM/localStorage/Canvas 的最小 stub，把 `physics.js / track.js / race.js /
ai.js / storage.js / tests.js / game.js` 逐一 `require` 進來實際執行、
驅動數千至數萬個模擬 tick 後驗證，而不是只靠人工讀碼判斷。

但本開發環境**沒有可用的真實瀏覽器**（無 Chromium/Firefox，也沒有
Node 版 `jsdom`/`puppeteer` 可用），因此以下項目**只完成了程式碼撰寫與
無頭邏輯驗證，尚未實際在瀏覽器畫面中肉眼確認**，請你在打開 `index.html`
時協助確認：

- Canvas 實際繪製結果的視覺正確性（賽道/車輛/HUD 版面是否美觀、無重疊、
  顏色是否清楚可辨）。
- 追蹤攝影機／全賽道攝影機切換後畫面是否符合預期。
- 瀏覽器 Console 是否有任何執行期錯誤或警告（尤其是 Canvas 2D API 在
  不同瀏覽器的 `roundRect` 相容性 —— 程式已對不支援 `roundRect` 的瀏覽器
  做了 `ctx.rect` 後備處理，但仍建議實機確認一次）。
- 實際鍵盤操作的手感（轉向/煞車力道是否符合你的預期，可能需要依實際
  試玩微調 `physics.js` 內的數值常數）。
- `localStorage` 在你實際瀏覽器的 `file://` 或 `http://localhost` 下是否
  正常寫入/讀取（不同瀏覽器對 `file://` 下 `localStorage` 的支援程度不同，
  這也是 README 建議可用 `python -m http.server` 開啟的原因）。
- 不同螢幕尺寸/視窗縮放下 HUD 版面與 Canvas 縮放的實際觀感。

請勿在未實際開啟瀏覽器確認上述項目前，假設它們必然正常運作。
