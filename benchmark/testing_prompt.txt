請在目前的空白目錄中，從零開發一個完整、可實際遊玩的瀏覽器遊戲：

Apex Sprint Challenge

這是一款俯視角的 F1 風格開輪式賽車遊戲。玩家要與三名 AI 對手進行短程競賽，管理煞車點、輪胎、ERS、DRS 與進站策略。

請直接建立完整可執行成果，不要只提供設計文件、片段程式碼或示意畫面。

==================================================
一、技術限制
==================================================

只能使用：

- HTML
- CSS
- 原生 JavaScript
- HTML Canvas
- Browser API

禁止使用：

- npm
- TypeScript
- React、Vue 或其他框架
- 遊戲引擎或物理引擎
- 外部 JavaScript 函式庫
- CDN
- 後端服務
- 外部圖片、字型或聲音素材

所有畫面、車輛、賽道、標誌與特效都必須使用 Canvas 或 CSS 繪製。

遊戲必須優先支援直接開啟：

index.html

如果瀏覽器限制導致部分功能無法透過 file:// 執行，可以在 README 補充：

python -m http.server 8000

但不可要求 npm install 或建置流程。

不得使用真實 F1 車隊 Logo 或商標。請使用虛構車隊與配色。

==================================================
三、基本遊戲
==================================================

建立一條虛構的封閉式賽道，至少包含：

- 一條長直線
- 一個高速彎
- 一個低速髮夾彎
- 一組連續彎
- Pit Lane
- Pit Box
- 起跑線
- 至少三個依序通過的檢查點
- 一段 DRS Zone
- 一個 DRS Detection Point

比賽包含：

- 玩家一輛車
- 三輛 AI 車
- 三圈比賽
- 五盞起跑燈
- 起跑倒數
- 完賽排名
- 最快圈
- 最終成績畫面

玩家控制：

- W 或方向鍵上：油門
- S 或方向鍵下：煞車
- A、D 或左右方向鍵：轉向
- Shift：使用 ERS
- F：啟用 DRS
- P：暫停或繼續
- R：重新開始
- C：切換追蹤攝影機與全賽道攝影機

當使用者正在 input、select、textarea 或 contenteditable 元素輸入時，不可誤觸遊戲快捷鍵。

==================================================
四、車輛物理
==================================================

不要求專業賽車物理引擎，但必須有一致且可理解的駕駛感。

至少實作：

- 油門加速
- 煞車減速
- 空氣阻力
- 滾動阻力
- 車速越高，轉向反應有所不同
- 基本側向抓地力
- 過度高速轉彎會產生滑動
- 草地抓地力較低
- 草地阻力較大
- 車輛與賽道護牆碰撞
- 玩家與 AI 車輛碰撞
- 碰撞後不可讓車輛永久卡住
- 車輛完全停止後仍可重新起步

物理更新必須使用固定時間步進，例如 60 Hz，不可直接把畫面幀率當成模擬速度。

當瀏覽器短暫卡頓時，不可讓車輛突然穿越整個賽道。

==================================================
五、圈數與比賽規則
==================================================

圈數不能只靠穿越終點線判定。

有效圈必須：

1. 依正確順序通過所有檢查點。
2. 最後穿越起跑線。
3. 不可倒車反覆穿越終點線增加圈數。

畫面顯示：

- 目前圈數
- 比賽總圈數
- 當前名次
- 當圈時間
- 最快圈
- 與前車的時間或距離差
- 比賽狀態

暫停時：

- 車輛停止更新
- AI 停止更新
- 比賽時間停止
- 圈速停止
- 胎溫、磨耗、油量與 ERS 狀態停止變化

重新開始時必須完整恢復：

- 車輛位置
- 車速和方向
- 圈數
- 檢查點
- 排名
- 比賽時間
- 輪胎
- 油量
- ERS
- DRS
- AI 狀態
- 起跑燈

==================================================
六、輪胎、油量、ERS 與 DRS
==================================================

輪胎必須具有：

- 胎溫
- 磨耗百分比
- 抓地力係數

輪胎規則：

- 冷胎抓地力較差
- 正常工作溫度抓地力最好
- 過熱時抓地力下降
- 磨耗增加時抓地力下降
- 激烈轉彎、打滑與重煞會增加胎溫及磨耗
- 草地行駛會讓輪胎稍微降溫

油量規則：

- 油量隨時間降低
- 油量越少，車輛重量稍微降低
- 重量降低後，加速可以稍微改善
- 油量不得變成負數

ERS 規則：

- 按住 Shift 時提供額外加速
- 電量逐漸下降
- 煞車時可以少量回充
- ERS 電量不得低於 0 或高於 100
- 電量耗盡後不得繼續提供額外動力

DRS 規則：

- 只能在 DRS Zone 內使用
- 第一圈不可使用
- 玩家在 Detection Point 與前車距離夠近時，該圈可以使用
- 煞車或離開 DRS Zone 時自動關閉
- DRS 開啟時降低空氣阻力
- 不符合條件時按 F，畫面應顯示原因

DRS 規則可以採用距離判定，不必精確模擬真實 FIA 規章。

==================================================
七、Pit Stop
==================================================

玩家可以駛入 Pit Lane。

進站規則：

- Pit Lane 有速度限制
- 超速時顯示警告並增加時間處罰
- 玩家必須在 Pit Box 內停車
- 停妥一段時間後完成換胎
- 換胎後輪胎溫度降低
- 輪胎磨耗恢復
- 畫面顯示進站倒數
- 完成後可以駛離 Pit Lane

不得在正常賽道上按一個按鈕直接換胎。

AI 車輛可以自行決定是否進站，但至少要有一輛 AI 在輪胎磨耗過高時嘗試進站。

==================================================
八、AI 對手
==================================================

建立三名虛構 AI 車手。

AI 至少必須：

- 沿賽車線行駛
- 看向前方目標點
- 在直線加速
- 在彎道前煞車
- 依彎道急緩調整目標速度
- 駛出賽道後嘗試回到賽道
- 避免永久撞牆
- 能正常完成圈數
- 使用 ERS
- 符合條件時使用 DRS
- 輪胎磨耗過高時考慮進站

提供兩種難度：

- Normal
- Hard

Hard 模式可以：

- 煞車較晚
- 過彎速度較高
- 更積極使用 ERS

不得只用固定速度讓 AI 沿路徑瞬間移動。

AI 必須受到與玩家相同或相近的基本物理限制。

==================================================
九、Demo Race 自動展示模式
==================================================

這是必要功能，用來驗證整場比賽。

主選單加入：

Demo Race

啟動後：

- 玩家車也由 AI 控制
- 四輛車自動進行完整比賽
- 不需要使用者按鍵
- 所有圈數、排名、輪胎、ERS、DRS 和進站規則仍正常運作
- 最後必須顯示完賽結果
- Demo 模式可以暫停、繼續與重新開始

Demo Race 不可使用預先寫死的結果，也不可直接改變車輛位置來假裝比賽。

==================================================
十一、遙測與資料保存
==================================================

每隔固定時間記錄玩家：

- 時間
- 圈數
- X、Y 位置
- 速度
- 油門
- 煞車
- 轉向
- 胎溫
- 輪胎磨耗
- ERS
- DRS
- 是否在賽道上

提供 Export Telemetry 按鈕，將資料下載成 CSV。

使用 localStorage 保存：

- 最快圈
- 難度設定
- 音效或介面設定
- 最近一次比賽結果

localStorage 資料損壞時，遊戲不可崩潰，應恢復預設值並顯示提示。

==================================================
十二、瀏覽器測試
==================================================

建立 tests.html 和 tests.js。

開啟 tests.html 後，自動顯示：

- 每項測試名稱
- PASS 或 FAIL
- 實際錯誤訊息
- 通過數量
- 失敗數量

測試必須直接使用正式遊戲的函式或資料模型，不可重新寫一套假邏輯。

至少包含以下測試：

1. 未依序通過檢查點時，不可增加圈數。
2. 依序通過檢查點並穿越終點線時，圈數增加一次。
3. 倒車重複穿越終點線不能增加圈數。
4. 暫停後，比賽時間與車輛位置不再更新。
5. ERS 不會低於 0 或高於 100。
6. ERS 耗盡後不再提供額外動力。
7. 第一圈不可使用 DRS。
8. 離開 DRS Zone 後，DRS 自動關閉。
9. 輪胎磨耗增加時，抓地力應下降。
10. Pit Stop 完成後，輪胎磨耗恢復。
11. Reset 能恢復初始比賽狀態。
12. 排名計算不可以只比較圈數，還要考慮賽道進度。
13. 損壞的 localStorage 資料不會讓遊戲崩潰。
14. Demo Race 使用真實遊戲更新流程，而不是直接寫入結果。

不要建立：

- 永遠回傳 true 的測試
- 只測試常數的測試
- 沒有呼叫正式程式的測試
- 預先寫死 PASS 的測試

十四、完成條件
==================================================

只有真正達成以下條件後，才可以宣稱完成：

1. index.html 可以實際開啟。
2. 主選單可以開始正常比賽。
3. 玩家車可以加速、煞車與轉向。
4. AI 可以沿賽道完成有效圈。
5. 圈數與檢查點規則可以正常運作。
6. 暫停與重新開始可以正常運作。
7. ERS 可以使用、耗盡與回充。
8. DRS 受到區域與資格限制。
9. 輪胎溫度與磨耗會影響抓地力。
10. 玩家可以完成一次 Pit Stop。
11. Demo Race 可以自動進行。
12. tests.html 已實際執行。
13. Telemetry CSV 可以匯出。
14. 瀏覽器 Console 沒有未處理錯誤。
15. 無法驗證的功能必須明確列出，不能假設成功。

不要因為程式碼已經寫完，就宣稱遊戲可以正常運作。

最終回覆必須包含：

- 實作摘要
- 程式架構
- 建立或修改的檔案
- 實際執行的驗證
- tests.html 的測試結果
- Demo Race 的驗證結果
- 已知限制
- 未完成或無法驗證的項目

請現在開始檢查目前目錄、規劃並實作整個遊戲。
